# 开发者自查清单 — 先训练自己，再训练 AI

> 来自 Karpathy 哲学："AI 会放大你的工作方式。如果你自己都没想明白，AI 只会放大混乱。"

## 用 AI 写代码前，先问自己 5 个问题

### 1. 需求是否干净？
- [ ] 我用一句话能说清楚目标是什么吗？
- [ ] 我知道什么**不做**吗？（比"做什么"更重要）
- [ ] 验收标准是可测量的吗？（不是"看起来不错"）

### 2. 上下文是否充足？
- [ ] 我让 AI 先读了相关文件吗？
- [ ] 我告诉了 AI 项目的代码风格和约定吗？
- [ ] 我给了输入输出的具体格式吗？

### 3. 边界是否清晰？
- [ ] 我明确了哪些文件可以改、哪些绝对不能碰吗？
- [ ] 我知道改动的影响范围吗？
- [ ] 如果有数据库变更，有回滚方案吗？

### 4. 反馈窗口是否打开？
- [ ] 写完代码后有测试可以跑吗？
- [ ] 有 lint/类型检查可以自动验证吗？
- [ ] 有办法启动服务看实际效果吗？

### 5. 经验是否沉淀？
- [ ] 这次遇到的问题，以前遇到过吗？
- [ ] 如果是第三次遇到，我写 Skill 了吗？
- [ ] 这次学到的东西，下次怎么让 AI 自动知道？

## AI 输出后，必须做 3 件事

### 1. 逐行读 diff
```
不要只看 AI 说"我改好了"。
必须 git diff 逐行确认：
- 每一行改动都能追溯到你的请求吗？（Karpathy: Surgical Changes）
- 有没有"顺手"改的无关代码？
- 有没有删掉不该删的注释/代码？
```

### 2. 跑验证
```
- [ ] 类型检查通过？
- [ ] Lint 通过？
- [ ] 测试通过（新增 + 已有）？
- [ ] 手动启动服务验证？
```

### 3. 记录
```
- 如果 AI 搞砸了，在 /memories/repo/known-issues.md 记录
- 如果某个模式反复出现，考虑写成 Skill
- 如果某段 prompt 特别有效，保存为模板
```
